# Evaluation

## Client Feedback

After consulting my client, Mr Daniel Blanchard (maths and further maths teacher at Langley Grammar School), over a meeting in which I demonstrated the completed system, I have summarised his thoughts and feedback below:

> The system you have built is a genuinely impressive piece of software, and I can see real potential for it to support students who currently struggle with the cost and complexity of graphing calculators. The breadth of functionality — covering graphing, equation solving, statistics, and even SUVAT mechanics — goes well beyond what I initially expected, and the fact that it targets both a desktop and an embedded hardware platform shows real ambition.
>
> The graph plotter in particular stands out. The ability to plot up to four simultaneous functions in different colours, zoom and pan around the viewport, and trace along curves with live coordinate readout is exactly the kind of tool that helps students visualise transformations, roots, and intersections far more intuitively than working on paper. The equation solver is also very useful — being able to solve linear, quadratic, and simultaneous equations (including 3×3 systems) gives students instant verification of their working, which is invaluable during independent study.
>
> There are a few areas I think you could develop further in the future. Firstly, the system currently only supports real solutions; complex roots for quadratics (where the discriminant is negative) display "No real solutions" rather than the complex pair. For A-Level Further Maths, students do encounter complex numbers, so it would be a useful addition. Secondly, the Newton-Raphson solver only finds one root per call, dependent on the initial guess — it would be more powerful if it could automatically scan for multiple roots across a given interval and report all of them. Finally, a step-by-step solution mode — showing the working for a quadratic using the quadratic formula, or showing each Newton-Raphson iteration — would turn the calculator into a genuine learning tool rather than just an answer-checker, which is ultimately what students need most.

Taking my client's feedback into account, I have identified the following areas I would focus on in future development:

**Complex number support** was mentioned by my client as a gap in the current system. At present, if a quadratic such as $x^2 + 1 = 0$ is entered, the solver correctly identifies that there are no real solutions and displays an appropriate message. However, for A-Level Further Maths students, returning the complex pair $\pm i$ would be far more useful. Implementing this would require extending the expression evaluator to support complex arithmetic — `double` values would need to be replaced (or supplemented) with a `Complex` struct carrying a real and imaginary part, and all mathematical operations within `expr_eval.c` would need to be updated to propagate imaginary values correctly. This is a non-trivial change to the core evaluation engine but is the most educationally significant improvement I could make.

**Multiple root detection for Newton-Raphson** is the other solver limitation my client highlighted. The current implementation (`solver.c:73–105`) starts from a single user-supplied initial guess and converges to whichever root is nearest. For a function like $x^3 - x = 0$ (which has roots at $-1$, $0$, and $1$), the solver will only find one of the three. A future improvement would be to automatically sweep the initial guess across a defined interval, collect all distinct convergence points (filtering duplicates within a small tolerance), and display the full set. This would require running the Newton-Raphson routine multiple times in a loop and deduplicating results — computationally inexpensive given the algorithm's fast convergence.

**Step-by-step working display** is perhaps the most impactful improvement from a teaching perspective. Rather than displaying only the final answer, the system could show each stage of the solution — for example, the discriminant calculation and root extraction steps for a quadratic, or each iteration's $x$ value and $|f(x)|$ for Newton-Raphson. The Newton-Raphson solver already records the iteration count (`solver.c:87`), so extending this to log each $x_n$ value and display them in a scrollable results panel would be a natural first step. This aligns directly with my client's interview response to question 9, in which he specifically asked for step-by-step breakdowns as a desirable feature.

## Improvements I Could Make in the Future

Although I am satisfied with the system I have built, there are a number of improvements I would make given more time.

**Complex number support.** At the moment, if a quadratic with a negative discriminant such as $x^2 + 1 = 0$ is entered into the solver, it correctly displays "No real solutions". However for A-Level Further Maths students, it would be far more useful to return the complex pair $\pm i$ instead. To implement this, the expression evaluator would need to be extended to support complex arithmetic — `double` values would need to be supplemented with a `Complex` struct carrying a real and imaginary part, and all operations in `expr_eval.c` would need updating to propagate these correctly. This is the most educationally significant improvement I could make to the solver.

**Multiple root detection for Newton-Raphson.** Currently the Newton-Raphson implementation takes a single initial guess from the user and converges to whichever root is nearest. For a function like $x^3 - x = 0$, which has roots at $-1$, $0$, and $1$, only one root is returned depending on the starting value. A future version could sweep the initial guess across an interval automatically, run the algorithm from each starting point, and deduplicate the results within a small tolerance to return the full set of roots. This would make the numerical methods app significantly more powerful without much additional computation, since Newton-Raphson converges very quickly.

**Step-by-step working display.** Rather than just showing the final answer, the system could display each stage of a calculation — for example, showing the discriminant value and both root extractions for a quadratic, or printing each iteration's $x_n$ and $|f(x_n)|$ value for Newton-Raphson. The iteration count is already tracked in `numerical_methods.c`, so extending this to log intermediate values and display them in a scrollable panel would be a natural next step. This would turn the calculator from an answer-checker into a genuine learning tool, which is what my client specifically asked for in his feedback.

**Numerical methods visualisation.** The numerical methods app currently outputs only a numerical result. A more engaging implementation would animate the process visually — for integration, drawing each parabolic strip of Simpson's rule onto the graph canvas as the sum is computed, and for Newton-Raphson, plotting the tangent line at each iteration to show convergence graphically. This would require the numerical methods module to pass intermediate values back to the graph renderer at each step, but would make the mathematical process far more intuitive for students learning these methods for the first time.

**Curve fitting extended to polynomial fits and graph plotting.** The current curve fitting screen only supports linear regression and displays the result as a text equation. A natural extension would be to support quadratic and higher-order polynomial fits using least-squares regression, and to automatically plot the fitted curve onto the graph app alongside the original data points. This would make the feature directly useful for A-Level data analysis tasks, where students often need to identify the degree of a relationship between variables.

**ESP32 hardware improvements.** The physical device was successfully built and tested on an ESP32-S3 microcontroller driving a 320×240 ILI9341 TFT LCD, with a 9×6 button matrix connected via an MCP23017 I²C GPIO expander. While the hardware meets all the objectives set out in the requirements, there is room to improve the physical build quality — the current device is a hand-assembled prototype and does not have the same level of polish as a mass-produced calculator. Given more time, I would design a custom PCB to reduce wiring complexity, house the device in a 3D-printed enclosure, and improve battery life by optimising the display backlight and putting the ESP32 into a low-power sleep state when idle.

## What Went Well in the Project

Many aspects of the graphing calculator project were successful, and reflecting on the development process, several outcomes stand out as particularly strong.

**Effective alignment with client requirements.** The structured client interview provided a clear and detailed specification before any code was written. Every major feature in the final system traces directly to a stated client need. Graph plotting (objective 2.2) addresses the client's response to question 7 about the importance of visualising functions. The equation solver (objective 7) addresses his statement in question 2 that numerical equation solving is one of the most important features for the A-Level course. The statistics module (objective 5) addresses his response to question 8 about essential non-graph functions. The SUVAT mechanics solver goes beyond the original specification but was motivated by the client's teaching of A-Level Physics alongside Maths, and he responded positively to it in feedback.

**Cross-platform architecture.** The decision to separate platform-independent logic (`src/`) from platform-specific code (`desktop/`, `esp32/`) using a Hardware Abstraction Layer (`input_hal.h`) paid off significantly during development. All five application modules — Math, Graph, Stats, Solver, and Mechanics — were written once and compile without modification for both the desktop SDL2 environment and the ESP32 toolchain. This also meant that all testing and iteration could be done rapidly on desktop, where build times are short and debugging tools (GDB, sanitisers) are readily available, without sacrificing the ability to deploy to embedded hardware later.

**Recursive-descent expression parser.** The hand-written recursive-descent parser in `expr_eval.c` is the computational core of the entire system and it performs very well. It handles the full range of mathematical expressions required by the A-Level syllabus: standard arithmetic, exponentiation, trigonometric and logarithmic functions, factorial, floor/ceil/abs, the constants $\pi$ and $e$, variables $x$, $y$, $z$, implicit multiplication (e.g. `2pi`, `3sin(x)`), and unary operators. The algorithm runs in $O(n)$ time relative to expression length and uses no heap allocation — the C call stack itself encodes operator precedence. This made it both fast and reliable throughout development.

**Systematic iteration and debugging.** The iteration log (`iteration.md`) records nine complete design–test–fix cycles, each of which identified a real bug and resolved it with a targeted fix. Notable examples include: the button labels displaying SDL keycode names (fixed by adding fallback path search for the keymap config file and a CMake copy step); the double-character insertion bug in the graph formula editor and solver input fields (fixed by registering `LV_EVENT_PREPROCESS_KEY` handlers to consume special keys before LVGL passed them to the textarea); and the `sin(π)` floating-point artefact returning `1.22×10⁻¹⁶` instead of `0` (fixed by adding a post-evaluation snap in `expr_eval.c` that rounds results within `1e-10` of an integer). Each iteration left the system in a more reliable state than before.

**Custom Unicode font rendering.** A particularly successful technical solution was the generation of custom LVGL fonts (`font_math_12.c`, `font_math_14.c`) to cover the Greek Unicode range (U+0391–U+03C9) and mathematical symbols (`√`, `π`, `Σ`, `≤`, `≥`). The standard Montserrat font bundled with LVGL only covers basic Latin (U+0020–U+007F), so Greek letters such as $\sigma$, $\mu$, and $\theta$ in the statistics app were rendering as empty boxes. By generating custom glyph tables using the LVGL font converter and adding `UI_FONT_MATH` and `UI_FONT_MATH_SMALL` design tokens to `ui_common.h`, all mathematical symbols now render correctly across every screen in the application. This directly improves the professional appearance of the UI and makes results easier to read.

**Reusable UI component design.** The `ui_submenu.c` component provides a scrollable, highlighted vertical menu with standardised Up/Down/Enter/ESC/M key navigation. Rather than reimplementing menu logic separately in the Solver, Stats, and Settings apps (which would have resulted in three near-identical blocks of 100+ lines each), a single 144-line reusable component is shared across all three. This directly demonstrates the DRY (Don't Repeat Yourself) software engineering principle and ensured that navigation behaviour is consistent across the entire application — something my client specifically noted as important in his feedback on ease of use (question 5 of the interview).

**Client's specific feedback acted upon during development.** The client stated in question 2 that a colour screen and high-contrast visuals would make the calculator far easier to use in a classroom. This was one of the first design decisions made: each of the six main menu apps is assigned a distinct accent colour (blue for Math, red for Graph, green for Stats, orange for Solver, purple for Mechanics), and function curves in the graph plotter are plotted in four distinct colours (green, red, blue, orange) so that multiple plots can be distinguished at a glance. The client stated in question 6 that an intuitive interface is a priority; this motivated the implementation of smart deletion (the DEL key removes entire function names like `sin(` as a single unit rather than character by character) and the Ans recall button, both of which reduce the friction of entering complex expressions.
